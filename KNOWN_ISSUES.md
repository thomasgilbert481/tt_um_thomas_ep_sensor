## Root cause

Magic DRC v8.3.568 (precheck) flags 99 errors that come entirely from
**sky130 foundry-IP gencell variants** in our layout — not from any drawn
geometry of our analog circuit.

Verified by selectively deleting cell instances and re-running Magic:
- Removing 6 `sky130_fd_pr__res_xhigh_po_0p35_*` resistor instances and
  3 `sky130_fd_pr__cap_mim_m3_1_*` MIM cap instances (UBRWDH, XCEES9,
  CRXE5C) → **Magic DRC = 0**.

## Errors by rule

| Magic rule | Count (CI) | Source cells | Notes |
|---|---:|---|---|
| `Width of RPM/URPM < 1.27um (rpm.1)` | 22 | All 6 `res_xhigh_po_0p35` instances | URPM derived from `bloat-all uhrpoly xpc` per `sky130A.tech`; depends on internal poly geometry, not GDS marker |
| `MiM cap spacing to via2 < 0.1um (capm.8 - via2.4)` | 8 | `cap_mim_m3_1_UBRWDH` | Internal MIM cap layout |
| `MiM cap spacing to unrelated metal3 < 1.34um (capm.11)` | 70 | `cap_mim_m3_1_XCEES9`, `cap_mim_m3_1_CRXE5C` | Decap caps near M3 routing |

## Why this isn't a design defect

1. **Base foundry cells pass DRC standalone.** Loading
   `sky130_fd_pr__res_xhigh_po_0p35.gds` or
   `sky130_fd_pr__cap_mim_m3_1.gds` directly into magic gives 0 DRC errors.
   The errors only appear in the gencell-customized hash-named variants
   (`_YR6MM7`, `_LS96F9`, `_UBRWDH`, etc.) generated by older magic when
   the layout was constructed.

2. **KLayout DRC is clean** — both decks (foundry's `sky130A_mr.drc` with
   `feol=true / beol=true / offgrid=true`) report 0 violations.

3. **The chip's drawn analog circuit is verified** — Monte Carlo (500
   trials), post-layout robustness sim, and EP-slope verification all pass.

## What we tried

| Fix attempt | Result |
|---|---|
| Add wide GDS URPM markers (79/20) covering full cell width + 1 µm | No change. Magic computes URPM from poly bloat, not just GDS marker. |
| Switch from `res_xhigh_po` to `res_high_po` | Same `rpm.1` rule fires (it checks both RPM and URPM derived layers). |
| NMOS-as-resistor substitution | Top NMOS goes subthreshold when divider drop < Vt; would need PMOS-on-top + careful sizing — multi-day rework. |
| Notch metal at ua[5] site (TT pin sites) | Resolved Analog Pin Check ✅, no effect on Magic DRC. |
| Trim `res_xhigh_po_0p35` outer urpm tabs (the 0.75 µm wide tab regions) | Same rule still fires from poly-derived URPM inside. |
| Add full-cell-bbox URPM marker INSIDE each res cell definition | No effect — `mask-hints URPM` in `sky130A.tech` is hint-only, doesn't feed the rpm.1 width check. The rule fires on a derived layer computed entirely from the foundry cell's internal poly+xpc geometry. |
| Substitute digital pins (ui_in/uio_*) for bias | Not viable — TT IO ring quantizes digital pins to 0/VDPWR via level shifters; can't pass analog voltage. |
| Substitute n-well resistor (res_iso_pw, no poly markers) | Sheet R 3 kΩ/sq * min W 2.65 µm → 1.5 MΩ resistor needs ~1325 µm length, doesn't fit in remaining tile area. |

## Path forward (what we're asking TT)

External bias pins would clear all errors but **we have no spare analog
pin budget** (`analog_pins: 2`, all 6 ua sites are tile-spec'd, ua[2..7]
must remain unconnected per current `analog_pin_check`).

We've reached out to TT support (see `TT_OUTREACH.md`) asking for:

1. Known fix for sky130 `res_xhigh_po` gencell DRC under magic 8.3.568
2. Waiver mechanism for foundry-cell-internal DRC errors (analogous to
   the SRAM/IO carve-outs in `sky130A_mr.drc` for KLayout)
3. Recommended bias-network style for TT analog 2x2 tiles
4. Whether regenerating the gencells with current magic is expected to
   produce clean output (and the recommended flow if so)

## Reproducing locally

```bash
docker run --rm -v $(pwd):/work hpretl/iic-osic-tools:2025.07 bash -c '
  /foss/tools/magic/bin/magic -dnull -noconsole \
    -rcfile /foss/pdks/sky130A/libs.tech/magic/sky130A.magicrc \
    -T <<TCL
gds read /work/gds/tt_um_thomas_ep_sensor.gds
load tt_um_thomas_ep_sensor
select top cell
drc euclidean on
drc style drc(full)
drc check
drc catchup
puts "ERR=[drc list count total]"
quit -noprompt
TCL
'
```

Local magic 8.3.530 reports 24 unique violations (= same 99 edges, just
reported once each) — same root cause.
